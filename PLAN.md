# UTA Acting System — Design Plan

## Code Style

All Python in this repository follows the [Google Python Style Guide](https://google.github.io/styleguide/pyguide.html).
Key conventions: descriptive names (no acronyms or single-letter identifiers outside comprehensions),
module-level docstrings, type annotations on all public functions, and `snake_case` throughout.

## Context

The proposal defines a system to model theatrical characters as multi-scale latent dynamical systems and generate character-faithful improvised dialogue. The user has answered design questions:
- Creative elasticity over strict canon fidelity; playable objectives over literary truth
- Text-only input/output; one character's lines at inference time
- **Latent variable**: Option A (symbolic labels + text rationales) for MVP; migrate to Option B (probabilistic factor graph) as corpus grows
- **MVP corpus**: Chekhov, English-language translations
- **Evaluation**: LLM-as-judge (not human theater practitioners)
- **Beat segmentation**: LLM does one annotation pass per play; results cached as ground truth from that point on
- **Improv test**: cross-play pairing — Chekhov character meets Hamlet, both lines generated by the system

This plan synthesizes background research to inform the full system design.

**Recommended MVP character** (open question — pick one):

- **Vanya** (Uncle Vanya): frustrated idealist; self-destructive tactic distribution; strong arc across acts
- **Nina** (The Seagull): romantic idealism → disillusionment → hard-won clarity; clear arc and belief-state evolution

### What's Missing

No existing source has **beat segmentation** — this is a novel contribution of the system. All sources provide act/scene/utterance; beat is intermediate and will be produced by LLM.

### Parsing Pipeline

1. Download Chekhov plain text from Project Gutenberg; Hamlet TEI-XML from Folger/DraCor
2. For TEI: parse with `lxml`, extract `<sp>`, `<stage>`, `<div>` (act/scene), `<speaker>`
3. For plain text: regex-based parser for capitalized speaker names, stage directions in parentheses
4. Build `Play → Act → Scene → Utterance` objects
5. Beat segmentation: LLM pass over scene windows using theory-grounded prompt (see Pass 1 pipeline); cache results as `beats_ground_truth.json`

## 2. Latent Variable Formalism — Tradeoffs

This is the key architectural decision (Q4 from proposal). Three options:

### Option A: Symbolic Labels + Text Rationales

**What it is**: For each beat, an LLM produces structured JSON with categorical fields (tactic: "deflect", emotion: "anxious", etc.) plus a free-text rationale. States are human-readable strings.

**Pros**:
- Immediately interpretable by dramaturgs, directors, actors
- Easy to debug and inspect
- Can be used directly as LLM prompts for generation
- No training data needed; works with a single play
- Most aligned with the "literary analysis engine" use case

**Cons**:
- Categorical labels are coarse; don't capture continuous dynamics
- No principled uncertainty or multi-hypothesis representation
- Cannot be learned or refined through a probabilistic inference pass
- Transitions are discrete hops, not smooth trajectories

**Best for**: MVP; literary analysis output; small corpus (1 play)

---

### Option B: Probabilistic Latent States (Structured Factor Graph / HMM-like)

**What it is**: Each hidden state component (desire, affect, tactic, epistemic, social, defense) is a probability distribution over a discrete or continuous space. The full character state is a factored distribution. Inference is done via message passing or variational methods.

**Pros**:
- Principled multi-hypothesis representation (Hamlet *probably* wants X, with some probability of Y)
- Temporal consistency enforceable through smoothing
- Can propagate uncertainty across the play
- Naturally captures the "ambiguity is preserved" design goal
- State transitions are explicit and inspectable

**Cons**:
- Requires defining the state space formally (what are the possible tactics? how many emotion dimensions?)
- Inference is nontrivial to implement correctly
- LLM outputs need to be converted into likelihoods/probabilities — requires calibration
- At MVP scale (1 play), the prior distributions are hand-set or flat; learning requires a corpus
- Harder to use directly as generation prompts (need to convert back to natural language)

**Best for**: Phase B (corpus of plays); formal publication; rigorous evaluation

---

### Option C: Learned Latent Embeddings with Symbolic Heads

**What it is**: A neural model learns a continuous latent representation of character state. Symbolic heads (classifiers for tactic, emotion, etc.) are trained on top. The latent space is shaped by both the symbolic supervision and learned structure.

**Pros**:
- Most expressive; can capture nuances not expressible in discrete categories
- Generalizes across characters if trained on a large corpus
- Could learn character-specific subspaces automatically

**Cons**:
- Requires substantial training data (many annotated plays)
- Black-box: hard to interpret, hard to inject domain knowledge
- Not viable at all for MVP with 1 play
- Breaks the "understandable modular models" design goal

**Best for**: Phase C (only after many plays are annotated)

---

### Recommendation

**Start with Option A (symbolic + text rationales) for the MVP**. This is immediately actionable, matches the literary analysis use case, and produces human-readable character bibles. Design the schemas so that Option A outputs can be mechanically converted to Option B distributions later.

**Migration path**: When the corpus grows to 5+ plays, add Option B structure around Option A outputs — treat the LLM's categorical outputs as evidence in a factor graph and use belief propagation to enforce consistency. The categorical labels from Option A become the discrete states in Option B.

This gives you the "confident ML path" without premature statistical complexity.

## 3. Evaluation Design: Three-Tier Comparison

### Tier 1: Vanilla LLM Baseline

**Setup**: Zero-shot prompt: *"You are [Character]. The scene is: [context]. Say something."*

**Known failure modes** (from literature):
- Good at stylistic mimicry (tone, register) but poor at psychological consistency over multiple turns
- Ignores knowledge state: reveals secrets, forgets prior events
- Tactics are generic (LLM defaults to being helpful/agreeable, not dramatically interesting)
- No beat-level awareness; all lines feel equally weighted

**Metrics**: character classification accuracy (can a judge identify which character said it?); consistency score (do lines contradict canon facts?); style distance

---

### Tier 2: LLM + Text-Only Character Bible

**Setup**: System prompt includes the character bible (superobjective, recurring tactics, speech style, key relationships, known facts/secrets, arc by act).

**Expected improvement**: Better stylistic and factual consistency. Still limited by:
- Static bible doesn't track dynamic state (what changed in *this* scene)
- No mechanism to enforce tactic fidelity beat-by-beat
- No subtext structuring in prompt → surface-only generation

**Metrics**: same as Tier 1, plus: tactic agreement rate (does the generated line match the intended tactic?); knowledge leak rate (how often is forbidden knowledge revealed?)

---

### Tier 3: LLM + Full Reflection Loop

**Setup**: Structured latent state initialization → candidate generation → multi-axis scoring → targeted state-based feedback → revision (2–4 rounds).

**Reflection axes** (scored by a separate LLM critic or rule-based checker):
1. Voice fidelity: does this sound like the character?
2. Tactic fidelity: does the action verb match the intended tactic?
3. Knowledge fidelity: does the character reveal only what they should know?
4. Relationship fidelity: does status/warmth match the current relationship state?
5. Subtext richness: is something happening beneath the words?
6. Emotional transition plausibility: is the affect shift motivated?

**Feedback format** (targeted, not generic): *"This line is too direct; the character usually deflects before admitting vulnerability. Try again with [tactic: deflect]."*

---

### Training Data Contamination Risk

Hamlet and Cherry Orchard are among the most analyzed works in the Western literary canon. Any LLM used for generation or judging has almost certainly seen extensive commentary, character analyses, and even beat-level breakdowns of these plays during pretraining. This creates two methodological problems:

1. **Inflated baselines**: The vanilla LLM (Tier 1) and bible-augmented (Tier 2) conditions will perform artificially well on well-known characters, compressing the gap between tiers and obscuring the value added by the structured reflection loop (Tier 3).

2. **Unclear attribution**: When Tier 3 produces psychologically consistent Lopakhin or Hamlet lines, it is difficult to separate the system's contribution (state tracking, tactic enforcement, reflection) from the LLM's memorized knowledge of these characters.

#### Mitigation: Low-Familiarity Control Condition

To isolate the system's contribution, the evaluation must include at least one **low-familiarity** character that the LLM is unlikely to have deeply memorized. This serves as a control where the system's structured state representation is the primary source of character knowledge, not the LLM's pretraining.

**Candidate sources for low-familiarity characters** (pick one):

| Source | Example | Why it works |
|---|---|---|
| Minor Chekhov one-acts | Smirnov (*The Bear*), Lomov (*The Proposal*) | Same author/style as MVP corpus but far less analyzed; tests whether system generalizes within Chekhov |
| Obscure 19th-century drama | Characters from Turgenev's *A Month in the Country*, Ostrovsky's *The Storm* | Period-appropriate; minimal LLM exposure; available on Project Gutenberg |
| Synthetic character | A novel character with a hand-authored bible (no canonical source text) | Fully isolates system contribution; no possible leakage; tests Pass 2 in its purest form |

**Recommended approach**: Add one low-familiarity canonical character (e.g., Lomov from *The Proposal*) **and** one synthetic character to the evaluation. This yields a 2×3 matrix:

|  | High-familiarity (Lopakhin / Hamlet) | Low-familiarity (Lomov) | Synthetic (no source text) |
|---|---|---|---|
| **Tier 1** (vanilla) | Leakage baseline | True baseline | True baseline |
| **Tier 2** (+ bible) | Moderate signal | Clean signal | Clean signal |
| **Tier 3** (+ reflection) | Ambiguous signal | Clean signal | Cleanest signal |

The **delta between tiers on low-familiarity and synthetic characters** is the strongest evidence for the system's value. High-familiarity characters remain useful for validating Pass 1 extraction (since human readers can judge correctness) but are weaker evidence for Pass 2 generation quality.

#### Ablation: Knowledge Leakage Test

As a direct diagnostic, run Tier 1 (vanilla, no bible) on all characters and measure performance. If the vanilla LLM scores comparably to Tier 2 on Hamlet/Lopakhin but poorly on the low-familiarity character, that is direct evidence of contamination and validates the need for the control condition.

---

### LLM-as-Judge Evaluation Protocol

Use an LLM judge (e.g., Claude Opus or GPT-4o) to evaluate each generated line blind to condition. The judge is given the character bible and scene context, then asked to rate each axis:

| Dimension | Judge prompt | Scale |
|---|---|---|
| Recognizability | "Given this character bible, does this line sound like the character?" | 1–5 |
| Playability | "Does this line give the actor a clear, active objective to play?" | 1–5 |
| Tactic fidelity | "Does the character's action (what they're doing *to* the other person) match the specified tactic?" | 1–5 |
| Subtext | "Is there something happening beneath the surface of the words?" | 1–5 |
| Earned affect | "Is the emotional register appropriate and motivated by prior context?" | 1–5 |
| Knowledge fidelity | "Does the character reveal any information they should not yet know?" | pass/fail + explanation |
| Tactic identification | "Name the action verb describing what the character is doing." | open text |

Run A/B/C comparisons across ≥20 generated scenes per condition. For tactic identification, measure agreement rate between judge-identified action verb and the BeatState tactic label.

**Consistency check**: Run each judge prompt 3 times with temperature > 0 and report variance to assess reliability. Use an ensemble of 2 judge models where possible.

## 5. Full System Design

### Data Model (Pydantic schemas, Pass 1 outputs)

```
Play
  ├── WorldBible (era, genre, social norms, factual timeline)
  ├── CharacterBible[] (per character)
  │     ├── superobjective: str
  │     ├── wounds_fears_needs: str
  │     ├── recurring_tactics: list[str]       # action verbs
  │     ├── emotional_tendencies: dict
  │     ├── speech_style: str
  │     ├── defense_mechanisms: list[str]
  │     └── arc_by_scene: dict[scene_id → str]
  ├── Scene[]
  │     ├── SceneBible (pressure, what_changes, hidden_tensions)
  │     └── Beat[]
  │           ├── utterances: list[Utterance]
  │           └── BeatState[] (per character)
  │                 ├── desire_state: str
  │                 ├── affect_state: dict (valence, arousal, certainty, control, vulnerability)
  │                 ├── social_state: dict (status, warmth)
  │                 ├── epistemic_state: dict (known_facts, hidden_secrets)
  │                 ├── tactic_state: str        # action verb
  │                 └── defense_state: str
  └── RelationshipEdge[] (character_a, character_b, temperature_by_beat)
```

### Pass 1 Pipeline

```
1. Ingest        → Parse TEI/XML from DraCor; extract Utterance objects
2. Beat segment  → LLM sliding window over scene using theory-grounded prompt: "A beat ends when objective or tactic shifts"; cache as ground truth JSON
3. Extract       → Per beat, per character: LLM extracts BeatState (structured JSON)
4. Smooth        → Global arc refinement: LLM reads full character trajectory, flags inconsistencies
5. Build bibles  → Aggregate BeatStates into CharacterBible, SceneBible, WorldBible
6. Validate      → Canon contradiction checker; confidence flagging for ambiguous beats
```

### Pass 2 Inference Loop

```
Input:  new scene context + which character to generate for
Step 1: InitState  → project character bible + context into initial BeatState
Step 2: Generate   → LLM with structured BeatState as control prompt → candidate line
Step 3: Score      → LLM critic + rule-based checkers on 6 axes
Step 4: Feedback   → State-based targeted feedback if any axis below threshold
Step 5: Revise     → Regenerate (max 3 rounds)
Step 6: UpdateState → Advance BeatState for next turn
Output: line + updated state + debug trace
```

**Test set (Pass 1 eval)**: 10 scenes from the primary play; LLM judge rates coherence and psychological plausibility of extracted BeatStates

**Test set (Pass 2 eval)**: 5 novel improvised scenes in alternate contexts; 1 cross-play scene (Chekhov character + Hamlet in a new setting, both generated by system, no shared canonical history). Must include scenes for the low-familiarity and synthetic characters to control for training data contamination (see §3, Training Data Contamination Risk).

**3-tier comparison**: 20 scenes per condition × 3 conditions × 3 character familiarity levels (high / low / synthetic); LLM-as-judge on 7 dimensions. The primary evidence for the system's value comes from the delta between tiers on low-familiarity and synthetic characters.

**Knowledge leakage ablation**: Run Tier 1 (vanilla, no bible) on all characters; report scores to quantify how much the LLM already "knows" about each character from pretraining.

**Pass 1 deliverable**: CharacterBible, SceneBible, WorldBible, BeatState archive (Option A: symbolic + rationales)

**Pass 2 deliverable**: Reflection loop with 6-axis scoring; debug trace output; 3-tier evaluation report
